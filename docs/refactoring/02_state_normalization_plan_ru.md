# Детальное ТЗ: Шаг 2. Нормализация внутреннего state-слоя

Дата: 2026-03-15

Статус документа:
- draft for implementation

Связь с master roadmap:
- этот документ является детальным ТЗ для шага 2 из [00_full_refactoring_roadmap_ru.md](/srv/git_projects/Samovar/docs/refactoring/00_full_refactoring_roadmap_ru.md)

## 1. Назначение

После завершения structural cutover проект больше не зависит от крупных legacy-монолитов, но внутренняя state-модель все еще остается трудно читаемой:
- в коде используются магические числовые значения для `SamovarStatusInt`, `sam_command_sync`, `startval` и ряда связанных флагов;
- смысл части состояний известен только по косвенным проверкам в `if/else` и по старым привычкам;
- логика смены режима и reset-пайплайн размазаны по нескольким владельцам ответственности;
- связь между `Samovar_Mode`, `Samovar_CR_Mode`, runtime-подсистемами и UI/web/Lua side effects остается неявной.

Цель шага 2:
- сделать state-модель проекта явной, читаемой и проверяемой;
- сохранить все числовые значения, внешние контракты и runtime-поведение 1:1;
- убрать зависимость логики переходов от "голых чисел" и неявных допущений.

## 2. Обязательные инварианты шага

- Нельзя менять численные значения статусов, команд и режимов.
- Нельзя менять последовательность side effects при переключении режимов и reset runtime.
- Нельзя менять HTTP endpoints, JSON keys, Lua names и menu/web-контракты.
- Нельзя вводить shim, adapter, compat-layer или временные прокладки.
- Нельзя выносить бизнес-логику в новый "переходный" слой ради миграции.
- Каждый подшаг должен заканчиваться собираемым состоянием и доказуемыми проверками.

## 3. Что входит в шаг 2

- инвентаризация и фиксация всех используемых status codes, command codes и mode codes;
- введение именованных констант или enum там, где сейчас используются голые числа;
- явная таблица соответствий "режим -> runtime/page handler/surface";
- явная таблица соответствий "статус -> смысл -> владелец";
- фиксация side effects для reset-пайплайна и смены режима;
- обновление regression-тестов так, чтобы они проверяли семантику, а не случайную текстовую форму старого кода;
- compile/build verification на целевых env.

## 4. Что не входит в шаг 2

- смена поведения режимов;
- изменение текстового формата программ;
- переписывание codec-слоя режимов;
- redesign web/menu/Lua;
- изменение имен публичных web/Lua contracts;
- перенос state на новый runtime framework или внедрение новой state machine библиотеки.

## 5. Архитектурное решение шага

Рекомендуемый подход:
- не строить новый уровень абстракции "над проектом";
- нормализовать существующий state-слой в текущих владельцах ответственности;
- использовать именованные значения и компактные helper-таблицы только там, где они убирают неявность;
- держать source of truth ближе к владельцу runtime и mode orchestration, а не в отдельной миграционной прослойке.

Результат шага должен дать ответы на три вопроса прямо по коду:
- какое число что означает;
- какой режим к какой runtime-ветке относится;
- какой reset и какие side effects обязаны выполняться при смене режима.

## 6. Декомпозиция на подзадачи

### Подзадача 2.1. Инвентаризация state-кодов и baseline-фиксация

Цель:
- собрать полный текущий набор кодов и точек использования для `SamovarStatusInt`, `sam_command_sync`, `Samovar_Mode`, `Samovar_CR_Mode`, `startval` и связанных флагов.

Что сделать:
- найти все сравнения и присваивания по ключевым state-переменным;
- сгруппировать их по владельцам: mode runtime, web routes, menu actions, Lua, loop/orchestration;
- зафиксировать baseline-таблицу в документации.

Артефакт:
- документированная таблица "значение -> смысл -> где используется".

Критерии приемки:
- нет "неразобранных" чисел в ключевых state-переменных;
- baseline-таблица покрывает все реально используемые значения.

Проверки:
- regression-тест или doc-test, подтверждающий, что зафиксированные значения присутствуют в коде и не менялись.

### Подзадача 2.2. Именованные статусы и команды без смены значений

Цель:
- убрать голые числа из переходной логики и reset-проверок.

Что сделать:
- ввести именованные константы или enum для status codes и command codes;
- заменить прямые сравнения на именованные значения в runtime/web/menu/Lua точках, где это улучшает читаемость;
- не менять layout данных и не менять сами числа.

Ограничения:
- не использовать enum class там, где это ломает существующие Arduino/C-style вызовы или требует массовой несовместимой миграции;
- не разносить одинаковые значения по нескольким местам.

Критерии приемки:
- в ключевой переходной логике отсутствуют необъясненные literal codes;
- числовые значения сохранены 1:1.

Проверки:
- baseline regression на exact numeric values;
- сборка `pio run`;
- точечные behavioral harness-тесты на mode/status switching.

### Подзадача 2.3. Явное описание mode ownership и runtime routing

Цель:
- сделать явной связь между режимом и тем, какой runtime/UI/web surface ему соответствует.

Что сделать:
- собрать mode mapping: режим -> active page -> runtime handlers -> program defaults -> optional Lua script family;
- нормализовать текущие проверки по `Samovar_Mode` и `Samovar_CR_Mode`;
- убрать неявные ветки, где режим "подразумевается" без явной декларации.

Критерии приемки:
- для каждого режима существует одна читаемая точка описания ownership;
- `change_samovar_mode()` и связанные вызовы используют явную mode semantics.

Проверки:
- regression-тесты mode-switch invariants;
- menu/web/Lua integration smoke на существующих harness-тестах;
- build matrix как минимум `Samovar` и `lua_expander`.

### Подзадача 2.4. Формализация reset-пайплайна

Цель:
- явно зафиксировать, что именно считается "сбросом подсистем" при смене режима.

Что сделать:
- документировать и проверить полный reset pipeline:
  - runtime status reset;
  - power/stepper/I2C pump cleanup;
  - Lua state reset;
  - menu/UI reset;
  - sensor counters reset;
  - route rebinding;
  - profile save/load sequence;
- убрать неявные допущения из тестов и зафиксировать инвариант в одном месте.

Критерии приемки:
- reset pipeline описан и проверяется;
- отсутствуют зависшие состояния при последовательных переключениях режимов.

Проверки:
- behavioral harness на цепочку смен режимов;
- ручной smoke на железе: несколько последовательных переключений с возвратом в исходный режим.

### Подзадача 2.5. Финальная зачистка magic numbers в state-логике

Цель:
- завершить нормализацию без расползания локальных исключений.

Что сделать:
- дочистить остаточные literal comparisons в критической логике;
- оставить числа только там, где это часть низкоуровневого протокола, аппаратного ограничения или compile-time конфига, а не state semantics;
- обновить документацию по новым именованным значениям.

Критерии приемки:
- логика переходов больше не опирается на необъясненные literal codes;
- численные значения по-прежнему совпадают с baseline.

Проверки:
- статический regression-тест на отсутствие key magic numbers в transition logic;
- сборка;
- повторный smoke-набор.

### Подзадача 2.6. Финальная интеграционная верификация шага

Цель:
- закрыть step gate перед переходом к шагу 3.

Что сделать:
- прогнать полный regression-набор;
- прогнать build matrix по основным env;
- обновить финальный итоговый документ шага 2 с перечислением подтвержденных инвариантов.

Критерии приемки:
- все шаги 2.1-2.5 подтверждены evidence;
- state-модель читается через именованные значения и mapping, а не через поиск по голым числам.

## 7. Требования к доказательствам

Для каждой подзадачи обязательно фиксировать evidence в трех формах:
- код: какие файлы изменены и где теперь находится source of truth;
- тест: какой regression/harness подтверждает инвариант;
- сборка: какой `pio run` подтверждает, что шаг не сломал проект.

Если хотя бы по одному инварианту нет достаточного evidence, подзадача считается `not_done`.

## 8. Рекомендуемые тестовые пакеты шага

Минимальный обязательный набор:
- tests на baseline numeric contracts для state/status/mode values;
- behavioral harness на `change_samovar_mode()` и полный reset pipeline;
- tests на web setup/save mode switching;
- tests на menu integration по смене режима;
- tests на Lua runtime mode-selection там, где задействован `USE_LUA`.

Минимальный build-набор:
- `pio run -e Samovar`
- `pio run -e lua_expander`

Желательный расширенный build-набор:
- `pio run -e Samovar_s3`
- `pio run -e nbk_pressure`
- `pio run -e no_power`

Ручной smoke после интеграции:
- последовательное переключение всех режимов на железе;
- запуск и остановка хотя бы одного режима после смены;
- проверка, что web/menu отображают правильный runtime;
- проверка, что нет зависших флагов pause/power/program state.

## 9. Риски шага

Основные риски:
- под видом "нормализации" случайно изменить поведение;
- растащить state-константы по нескольким местам вместо одного owner;
- зафиксировать не фактический baseline, а желаемый смысл;
- сломать Lua/web/menu ветки, которые используют те же state values косвенно;
- ввести новую прослойку, которая переживет миграцию и станет новым legacy.

Способ снижения рисков:
- сначала фиксировать baseline тестами, затем переименовывать;
- менять по одной группе значений за раз;
- после каждой подзадачи держать проект в собираемом состоянии;
- не делать "большой архитектурный прыжок" одним коммитом.

## 10. Gate завершения шага 2

Шаг 2 считается завершенным только если одновременно выполнено все ниже:
- все ключевые status/mode/command values зафиксированы и названы;
- transition logic не использует необъясненные literal numbers;
- mode ownership и reset pipeline читаются прямо из кода и подтверждены тестами;
- сборка проходит на обязательных env;
- ручной smoke не выявляет зависших состояний при переключении режимов;
- поведение прошивки и внешние контракты не изменены относительно baseline.

## 11. Формат постановки задач в рамках шага 2

Каждую реализацию внутри шага 2 ставить как отдельную задачу с единым шаблоном:
- цель подзадачи;
- конкретная область кода;
- acceptance criteria;
- список обязательных regression/build checks;
- явные ограничения "что нельзя менять";
- Definition of Done с требованием evidence.

Рекомендуемые task ids:
- `t20_1_state_code_inventory`
- `t20_2_named_status_codes`
- `t20_3_mode_mapping_normalization`
- `t20_4_reset_pipeline_invariant`
- `t20_5_magic_number_cleanup`
- `t20_6_state_layer_final_verification`

## 12. Рекомендуемый следующий исполнимый task

Следующим открывать:
- `t20_1_state_code_inventory`

Почему:
- без полной baseline-инвентаризации легко переименовать не то, что реально является state contract;
- это наименее рискованный старт шага 2;
- он даст опорную таблицу для всех остальных подзадач.
