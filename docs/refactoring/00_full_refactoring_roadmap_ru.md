# Верхнеуровневый план полного рефакторинга Samovar

Дата: 2026-03-09

Назначение документа: зафиксировать полную последовательность рефакторинга проекта. Этот документ является master roadmap. Детальные ТЗ по отдельным шагам лежат рядом и являются дочерними документами.

## 1. Главная цель

Полный рефакторинг должен привести проект к ясной модульной структуре без legacy-монолитов, без shim и без дублирования кода, при этом поведение прошивки на всех поддерживаемых режимах должно остаться прежним.

## 2. Общие правила для всех шагов

- Каждый шаг должен заканчиваться собираемым состоянием проекта.
- Каждый шаг должен завершаться удалением legacy-остатков в своей области, а не переносом "на потом".
- Нельзя вводить новые архитектурные слои ради миграции.
- Нельзя менять внешний контракт без отдельного решения.
- Следующий шаг нельзя начинать, пока не выполнены gate-условия предыдущего.

## 3. Шаги полного рефакторинга

### Шаг 1. Structural cutover

Цель:
- разобрать текущие mixed-файлы по владельцам ответственности;
- убрать главные legacy-монолиты;
- получить читаемую структуру модулей без изменения логики.

Детальный документ:
- [01_structural_refactor_cutover_plan_ru.md](/srv/git_projects/Samovar/docs/refactoring/01_structural_refactor_cutover_plan_ru.md)

Ожидаемый результат:
- `logic.h`, `WebServer.ino`, `lua.h`, `FS.ino`, `NVS_Manager.ino`, `Menu.ino`, `sensorinit.h` больше не существуют как mixed-монолиты;
- режимы и общие сервисы разложены по каталогам;
- проект все еще работает по старым контрактам.

Gate перехода к шагу 2:
- structural cutover завершен полностью;
- сборка проходит;
- ручные smoke-сценарии по всем режимам пройдены;
- в дереве не осталось временных compatibility helpers.

### Шаг 2. Нормализация внутреннего state-слоя

Цель:
- убрать магические числа и неявные связи между `SamovarStatusInt`, `startval`, `Samovar_Mode`, `Samovar_CR_Mode`;
- сделать state-модель читаемой без изменения значений и логики переходов.

Детальный документ:
- [02_state_normalization_plan_ru.md](/srv/git_projects/Samovar/docs/refactoring/02_state_normalization_plan_ru.md)

Что входит:
- именованные константы или enum для status codes и command codes;
- явная таблица соответствий "статус -> смысл";
- явная таблица соответствий "режим -> runtime entry points";
- документирование side effects по основным переходам.

Что не входит:
- изменение числовых значений;
- изменение последовательности переходов;
- изменение HTTP/Lua contracts.

Gate перехода к шагу 3:
- код больше не опирается на голые числа в логике переходов;
- все численные значения сохранены 1:1.

### Шаг 3. Нормализация программ режимов

Цель:
- сделать внутреннюю модель программ читаемой и обслуживаемой, сохранив внешние текстовые форматы.

Детальный документ:
- [03_program_codec_normalization_plan_ru.md](/srv/git_projects/Samovar/docs/refactoring/03_program_codec_normalization_plan_ru.md)

Что входит:
- финальная зачистка mode-specific codec modules;
- единый набор validate helpers;
- документация форматов rect/dist/beer/nbk;
- устранение дублирования в run/set/get program коде там, где это можно сделать без миграционных слоев.

Что не входит:
- смена текстового формата строк программ;
- смена HTTP payload для программ.

Gate перехода к шагу 4:
- каждый режим имеет изолированный codec;
- формат программ не изменился;
- ручная загрузка программ из web продолжает работать без изменений.

### Шаг 4. Консолидация UI и transport-слоя

Цель:
- убрать дублирование между web, menu и Lua surfaces там, где они обращаются к одному и тому же runtime;
- сделать frontend и transport code проще без изменения контрактов.

Что входит:
- дедупликация `data/*.htm`;
- вынос общего JS-каркаса;
- финальная зачистка web route modules;
- финальная зачистка menu actions и Lua bindings.

Что не входит:
- redesign UI;
- изменение endpoints и JSON keys без отдельного решения.

Gate перехода к шагу 5:
- frontend не содержит крупного copy-paste каркаса;
- web/menu/Lua не дублируют одну и ту же бизнес-логику.

### Шаг 5. Эксплуатационная и инженерная доводка

Цель:
- закрепить результат рефакторинга так, чтобы проект можно было дальше развивать без возврата к legacy.

Что входит:
- архитектурная документация по новым каталогам;
- build и smoke-checklists;
- cleanup неиспользуемых include и dead code;
- финальная проверка naming, ownership и compile hygiene.

Итоговый результат:
- проект читается по структуре каталогов;
- новые изменения можно делать в пределах владельца ответственности без повторного расползания монолитов.

## 4. Порядок исполнения

- Шаг 1 обязателен и является стартовой точкой всего рефакторинга.
- Шаги 2-5 не начинаются частично "вдогонку" до завершения шага 1.
- Каждый следующий шаг открывается только после закрытия gate-проверок предыдущего.

## 5. Признаки срыва плана

- Внутри любого шага появляются "временные" adapters или wrappers.
- На одном шаге начинается перепроектирование поведения вместо структурного переноса.
- Refactor branch перестает собираться на длительное время.
- На шаге 1 начинается изменение внешнего контракта.
- После завершения шага legacy-монолит остается в дереве как пустая оболочка.

## 6. Definition of done для полного рефакторинга

Полный рефакторинг завершен только если:
- structural cutover завершен;
- внутренний state-слой нормализован;
- программы режимов изолированы и документированы;
- web/menu/Lua/frontend не содержат крупного дублирования;
- в проекте нет shim, adapter и compatibility layers;
- поведение прошивки не изменилось относительно baseline.
